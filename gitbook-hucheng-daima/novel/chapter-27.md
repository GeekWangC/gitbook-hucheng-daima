# 第二十七章 流量洪峰

[目录](../SUMMARY.md) · [术语表](glossary.md) · [时间线](timeline.md)

某个平常的周三，开源社区里一篇文章把他们的组件库顶上了热点。流量像潮水一样涌进来，下载曲线在面板上拔地而起。

“瓶颈在静态资源分发。”小齐——或者说那位新 ID——迅速定位，“部分地区命中率低。”

“把多地域的缓存预热打开。”陈砺说，“同时在 README 顶部加告示，请大家稍后再试。”

他们切换到带有更优 Anycast 的 CDN，将大文件拆分为可并行下载的若干块，打上了长缓存版本号；对 API 层加了轻量的速率限制，防止爬虫把小水渠挤断。

“我们也可以请社区帮忙。”小齐在 Discussion 里发帖，征集镜像源与网络反馈。几个外部开发者自发地提了 PR，修复了一个边角的兼容问题，还补了一段“离线安装”的说明。

屏幕上的曲线慢慢降了下来，像潮水退去后平整的沙滩。王序感叹：“这就是开源。世界在帮我们把系统撑大。”

夜里，陈砺把“流量洪峰”的复盘写成了案例，放进文档站的“运营事件库”。他知道，这是另一个层面的可观测与秩序。

第二天他收到了江牧的消息：“看到了你们的处理，干净漂亮。我们之前也遇到过类似情况，Anycast 在某些地区还是会抖，建议你们加一条‘自我诊断页’。”

“谢谢。”陈砺回，“愿意把你们的经验写一段给社区吗？”

几天后，仓库里多了一篇短文：《如何为用户提供自诊断》。页脚署名：来自友军的建议。开源的意义，也在于把竞争变成共同体的成长。
